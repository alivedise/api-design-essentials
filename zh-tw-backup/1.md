# [ADP-1] ADP 目的

## 目標

- 本文件的目標之一是建立一個標準，使公司內的每個 API 設計者都能創建出看起來像是同一個人設計的 API。
- 確保所有 API 設計都遵循統一的最佳實踐和指南，促進所有項目的一致性和質量。
- 通過提供一個共同的語言和框架來促進團隊之間的更好合作，使跨不同業務單位的 API 更容易理解和集成。
- 確保 API 設計優先考慮開發者體驗和最終用戶體驗，從而開發出更健壯和用戶友好的應用程序。

## 原則

- **產品無關性**：這些原則設計為產品無關性，可應用於公司內不同業務單位的每個產品、項目或應用程序。
- **規則和建議結構**：每個原則可能包含一個或多個規則或設計建議。
- **RFC 要求關鍵字**：每個規則必須使用大寫的 `MUST`、`MUST NOT`、`SHOULD`、`SHOULD NOT`、`MAY` 或 `MAY NOT`，遵循 [RFC 2119 要求關鍵字](https://datatracker.ietf.org/doc/html/rfc2119)。

## 描述

- **關鍵字的包含**：包含這些大寫關鍵字的描述被視為規則的一部分。
- **複雜規則**：如果一個規則需要超過一句話的描述，則應成為一個新的 ADP 頁面，並在 markdown 的 front matter 區域中記錄唯一 ID。此 ID 用於網頁路由和開發者之間的溝通。<aside> 💡 `遞增編號基於 ID` 的命名約定比 UUID 更優，因為這種方法簡化了標識符的追蹤和管理，使其更易於理解和維護。</aside>
    
    ```markdown
    ---
    id: 1
    ---
    
    # 必須將開發者體驗視為與用戶體驗同樣重要
    ## 概述
    ...
    ```
    

## 違反規則

- **指定違反**：如果需要違反某些規則，您 [可以指定違反的規則](757.md)。
    
    <aside>
    💡 如果有合理的違反理由，您應該向該原則提供反饋，以便考慮對整個公司的變更。
    
    </aside>
    

## 草案規則

- **內部 API 入口**：有一個內部 API 入口，用於應用、問題和事件註冊，靈感來自 IANA 註冊。
- **草案狀態**：在規則前使用 `[DRAFT]` 表示該規則尚不穩定。

## 提出新的 ADP

- **格式**：使用 markdown 格式。
- **提交**：在 ADP 網站的倉庫中提交 pull request。（注意：這可能是鏡像倉庫。查找上游倉庫）。
