# [ADP-48] API 健壯性

## 概述

API 健壯性指的是 API 處理廣泛輸入和使用場景而不發生故障的能力。在設計 API 時，一個關鍵考慮是採用嚴格或寬鬆的方法來進行輸入驗證、錯誤處理和整體設計理念。本文討論了這兩種方法的優缺點，並提供了示例來說明每種方法。

## 指導原則

- 在同一應用程序中，應一致地做出 API 健壯性決策。

## 嚴格的 API 指南

### 定義

嚴格的 API 強制執行嚴格的驗證和遵守定義的標準。每個請求必須精確符合預期格式，任何偏差都會導致錯誤。

### 優點

1. **一致性**：確保所有輸入標準化，減少意外行為的風險。
2. **安全性**：通過嚴格驗證所有輸入，保護 API 免受格式錯誤的請求和潛在攻擊。
3. **可預測性**：客戶端可以依賴明確定義的合同，使集成和調試更容易。

### 缺點

1. **僵化性**：對於輕微的偏差可能不太寬容，可能導致客戶端錯誤率較高。
2. **複雜性**：需要詳細的文檔和廣泛的驗證邏輯，增加開發和維護工作量。

### 示例

考慮一個用於創建用戶資料的 API，其具有以下嚴格要求：

- **端點**：`POST /users`
- **有效負載**：
    
    ```json
    {
        "username": "string (字母數字，3-20 個字符)",
        "email": "string (有效的電子郵件格式)",
        "age": "integer (18-100)"
    }
    ```
    

任何未能滿足這些標準的請求都會返回 `400 Bad Request` 響應，並附上詳細的錯誤信息：

```json
{
    "type": "https://example.com/probs/invalid-input",
    "title": "Invalid input",
    "status": 400,
    "detail": "The 'username' field must be alphanumeric and between 3 and 20 characters."
}
寬鬆的 API 指南
定義
寬鬆的 API 允許更大的靈活性，容忍偏離預期輸入格式的情況，通常實施默認值或寬容處理輕微錯誤。

優點
靈活性：適應更廣泛的輸入，減少客戶端錯誤的可能性。
易用性：簡化客戶端集成，因為輕微的錯誤不太可能導致錯誤。
用戶體驗：通過優雅地處理輕微問題，提供更順暢的用戶體驗。
缺點
不一致性：可能導致輸入格式的變異，降低 API 的可預測性。
安全風險：更寬容的輸入驗證可能使 API 暴露於安全漏洞。
處理複雜性：需要健壯的錯誤處理和默認機制，這可能實施起來較為複雜。
示例
對於同一用戶資料創建 API 採取寬鬆的方法：

端點：POST /users

有效負載：

json
複製程式碼
{
    "username": "string (可選)",
    "email": "string (可選，有效的電子郵件格式)",
    "age": "integer (可選，默認為 18)"
}
如果客戶端省略了 username 或 email 字段，API 仍會創建用戶資料，並在必要時分配默認值：

json
複製程式碼
{
    "id": "12345",
    "username": "defaultUser",
    "email": "user@example.com",
    "age": 18
}
平衡嚴格性和寬鬆性
最佳實踐
關鍵字段：對影響核心功能或安全性的關鍵字段應採用嚴格驗證，例如認證令牌或財務數據。
可選字段：對可選字段允許寬鬆處理，提供默認值或在適當時推斷值。
錯誤處理：實施全面的錯誤處理和日誌記錄，以監控和解決由寬鬆輸入引起的潛在問題。
文檔：明確記錄預期的輸入格式、驗證規則和任何寬鬆行為，指導客戶端正確使用 API。
示例
用戶資料創建的平衡方法可能涉及對 email 字段進行嚴格驗證（以確保有效的通信渠道）並寬鬆處理 username 和 age 字段：

端點：POST /users

有效負載：

json
複製程式碼
{
    "username": "string (可選，字母數字，3-20 個字符)",
    "email": "string (必需，有效的電子郵件格式)",
    "age": "integer (可選，默認為 18)"
}
響應將反映嚴格和寬鬆處理的結合：

嚴格錯誤：

json
複製程式碼
{
    "type": "https://example.com/probs/invalid-input",
    "title": "Invalid input",
    "status": 400,
    "detail": "The 'email' field must be a valid email format."
}
寬鬆處理：

json
複製程式碼
{
    "id": "12345",
    "username": "defaultUser",
    "email": "user@example.com",
    "age": 18
}
結論
在嚴格和寬鬆的 API 指南之間做出決定取決於具體的使用情況、安全需求和用戶體驗考量。通過仔細平衡這些方法，開發人員可以創建既可靠又靈活的 API，提升服務的整體可用性和安全性。
