# [ADP-403] Problem Type Design

According to RFC9457, we come up with the following design principles for a new problem type

## Guidance

- MUST be human readable uri reference.
- SHOULD use relative uri reference if you aren’t sure where to publish the problems.

## Design Flow

- Look around at current common valid HTTP Status.
- If there’s a status 100% match your case, use it with `type: about:blank` .
- If you need to explain a bit more about the error, use `detail` field.
- If you cannot find out a matching status, go to pre-defined problem-type list page and find out if there’s a suitable one.
- If you need to explain a bit more about the error, use `detail` field.
- If there’s no problem-type meet your requirement, try to propose a new one; find the closest status number with it or use `400` / `500` at first.
- After a period of API design, go back to review the 400/500 errors and try to extract common problem types.
- The proposed new type MUST be common enough.
- Use errors field to describe the multiple error if you need.

Notice: there’s vague part between the RFC and the real world design. We tried to use `detail` and `error` to provide more information. If you still need to divide the `type` into more subtypes, here are some proposals.

### General Guidelines for Designing a Problem Type According to RFC 9457

1. **The problem type must be of a general nature and not bound to a specific product or application.**
    - This ensures that the problem type can be widely applicable and not limited to a particular context or use case.
2. **The problem type must be reused as much as possible.**
    - Reusing existing problem types promotes consistency and reduces redundancy in problem type definitions.
3. **If a problem can be fully described by an existing HTTP status code (with details), there is no need to create a new type (use `about:blank`).**
    - This encourages leveraging the standard HTTP status codes and the `about:blank` type when they suffice to describe the problem, avoiding unnecessary creation of new problem types.

### Example Explanation

**Scenario:** You want to define a problem type for an "Unauthorized Access" issue in your API.

**Step-by-Step Process:**

1. **General Nature:**
    - The problem type for "Unauthorized Access" should not include any product-specific or application-specific details. It should simply describe the nature of the problem in a general way, such as "unauthorized".
2. **Reuse Existing Types:**
    - Before creating a new problem type, check if there is an existing problem type that can be reused. For instance, if there is already a problem type defined for authentication issues, you might use that.
3. **Use Existing HTTP Status Code:**
    - Check if the problem can be fully described by an existing HTTP status code. In this case, HTTP status code 401 ("Unauthorized") can describe the issue. Therefore, you might use `about:blank` and include details in the response body.

### Example Implementation:

### 1. Using a general problem type:

```json
{
  "type": "https://example.com/probs/authentication",
  "title": "Unauthorized Access",
  "status": 401,
  "detail": "You do not have the necessary permissions to access this resource.",
  "instance": "/login"
}
```

### 2. Reusing an existing type:

If there is an existing problem type for authentication issues, reuse it:

```json
{
  "type": "https://example.com/probs/authentication",
  "title": "Unauthorized",
  "status": 401,
  "detail": "Invalid credentials provided.",
  "instance": "/login"
}
```

### 3. Using `about:blank` or the original error text in kebab-case when appropriate:

If the issue can be fully described by the status code and a detail message:

```json
{
  "type": "about:blank", // unauthorized
  "title": "Unauthorized",
  "status": 401,
  "detail": "Authentication required to access this resource.",
  "instance": "/login"
}
```
