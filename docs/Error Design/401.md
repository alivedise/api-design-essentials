# [ADP-401] HTTP Problem Basics

## Overview

This document outlines the requirements and guidelines for ensuring compatibility with HTTP Problem for error responses. By adhering to these standards, we can provide consistent and meaningful error information to clients.

It’s obvious that HTTP Status for errors(4xx, 5xx) cannot provide enough information for the API invoker to understand what happens exactly. RFC 9457 provides a standard way to describe the errors.

## Guidance

- MUST be compatible with HTTP Problem for error response
- MUST return JSON for errors(This follows MUST return JSON for API results)
- MUST return `Content-Type: application/problem+json` for errors
    
    
    | property | description | required | sample |
    | --- | --- | --- | --- |
    | type(string, uri-reference) | MUST be an URI.
    
    Ideally this SHOULD point to a real web site to describe the type with more information, as RFC recommended.
    
    If we are not able to provide the absolute URI at the moment we introduce the problem type, use relative URI is acceptable. | YES | https://api.sample.com/problems/validation-error
    
    /problems/validation-error
    
    /problems/{problem-type} |
    | status(number) | MUST be the same as response status code; i.e., you MUST NOT replace this with your own error code system. | YES | 500 |
    | title(string) | Human readable description to the type. It SHOULD be 1-to-1 mapping with type field. | YES | “Validation error” |
    | instance(string, uri-reference) | The uri where the problem occurs. | NO | /resources/{resource-id} |
    | detail(string) | Human readable description for the specific problem; this is expected to be instance-specific. | NO |  |
    
    ```jsx
    500
    Content-Type: application/problem+json
    
    {
       type: '/problems/predefined-type',
       status: 500,
       title: 'human-readable error information in short'
    }
    ```
    
- SHOULD reuse existing problem type if possible
    - [DRAFT] SHOULD register problem type in internal problem-type registration page.
    - Problem type naming MUST follow URI convention(kebab-case)
    - Problem type MUST be product-agnostic and business-model-agnostic
- MUST provide error object info array for validation errors(in errors because it’s expected to be plural)
    
    
    | property(type, format) | description | required | example |
    | --- | --- | --- | --- |
    | detail(string) | A long description for current error instance | true |  |
    | path(string) | json pointer of the validation error if in body; key name if in header/query; relative uri reference if in path
    json pointer: ‘#/key1/key2’ | true |  |
    | in(string;enum) | one of ‘header’, ‘body’, ‘query’, ‘path’ being aligned to OpenAPI in https://swagger.io/docs/specification/describing-parameters/ | true |  |
    | details(array, optional) | If the errors are multiple in the path, need to provide the details object with { key, message}. | false |  |
    
    ```jsx
    HTTP/1.1 422 Unprocessable Content
    Content-Type: application/problem+json
    Content-Language: en
    
    {
      "type": "https://example.net/validation-error",
      "title": "Your request is not valid.",
      "status": 422,
      "errors": [
        {
          "detail": "must be a positive integer",
          "target": "#/age"
        },
        {
          "detail": "must be 'green', 'red' or 'blue'",
          "target": "#/profile/color"
        }
      ]
    }
    ```
    
    <aside>
    💡 This is the original example from RFC9457. We’d change some field for the errors design.
    
    </aside>
    
- MAY add extra field to expand the problem JSON if you really need another information
    
    ```jsx
    {
      "type": "https://example.net/validation-error",
      "title": "Your request is not valid.",
      "extraInfo": "...."
    }
    ```
    

## Design Thinking

Many websites use a custom error system to describe a custom error other than http status code, such as an additional number based code. However, the user would still need to query the meaning of the code from internet because there’s no standard across different applications.

### Innovated problem type default set

Since RFC 9457 does not provide more problem type samples, we innovated some pre-designed problem types here(would be listed in internal problem type registration page later), according to https://www.rfc-editor.org/rfc/rfc9457.html#name-defining-new-problem-types.

- **Validation Error**
    
    HTTP 422 is Unprocessed Content; we baked a new error type `validation-error` because it’s common to point out the client’s request is having issues.
    
    ```json
    {
      "type": "https://example.com/problems/validation-error",
      "title": "Validation Error",
      "status": 422,
      "detail": "One or more fields are invalid.",
      "instance": "/errors/12345",
      "errors": [
        {
          "in": "body",
          "path": "#/json/pointer/to/path",
          "detail": "should be one of the following values...",
          "details": [{
            "key": "role",
            "message": "should be one of the following values:..."
          }]
        }
      ]
    }
    ```
    
- **Rate Limit Exceeded**
    
    HTTP 429 is Too Many Requests; we baked a new error type `rate-limit-exceeded` because it is obviously very common to limit the API usage across different products/applications.
    
    ```json
    HTTP/1.1 429 Too Many Requests
    Content-Type: application/problem+json
    Retry-After: 3600
    X-RateLimit-Limit: 1000
    X-RateLimit-Remaining: 0
    X-RateLimit-Reset: 1625247600
    
    {
      "type": "https://example.com/problems/rate-limit-exceeded",
      "title": "Rate Limit Exceeded",
      "status": 429,
      "detail": "You have exceeded the allowed number of requests. Please try again later.",
      "instance": "/errors/991122"
    }
    ```
    
- Payment related errors
    - Insufficient funds
        
        ```json
        HTTP/1.1 400 Bad Request
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "https://example.com/probs/insufficient-funds",
          "title": "Insufficient Funds",
          "status": 400,
          "detail": "Your account does not have enough funds to complete this transaction.",
          "instance": "/transactions/12345",
          "balance": 50.00,
          "requiredBalance": 100.00
        }
        ```
        
    - Transaction-limit-exceeded
        
        ```jsx
        HTTP/1.1 403 Forbidden
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "https://example.com/probs/transaction-limit-exceeded",
          "title": "Transaction Limit Exceeded",
          "status": 403,
          "detail": "You have exceeded your daily transaction limit.",
          "instance": "/transactions/67890",
          "transactionLimit": 1000.00,
          "attemptedAmount": 1200.00,
          "resetTime": "2024-07-16T00:00:00Z"
        }
        ```
        
- User permission related
    - Insufficient permissions
        
        ```jsx
        HTTP/1.1 403 Forbidden
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "https://example.com/probs/insufficient-permissions",
          "title": "Insufficient Permissions",
          "status": 403,
          "detail": "You do not have the necessary permissions to access this resource.",
          "instance": "/resources/abc123",
          "user_id": "user-12345",
          "required_permission": "admin"
        }
        ```
        
    - Account disabled(banned)
        
        ```jsx
        HTTP/1.1 403 Forbidden
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "https://example.com/probs/account-disabled",
          "title": "Account Disabled",
          "status": 403,
          "detail": "Your account has been disabled. Please contact support for assistance.",
          "instance": "/login",
          "userId": "user-67890",
          "supportContact": "support@example.com"
        }
        ```
        
    - Insufficient role
        
        ```jsx
        HTTP/1.1 403 Forbidden
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "https://example.com/probs/role-insufficient",
          "title": "Role Insufficient",
          "status": 403,
          "detail": "Your current role does not grant you access to this resource.",
          "instance": "/resources/xyz789",
          "user_id": "user-54321",
          "user_role": "basic",
          "required_role": "manager"
        }
        ```
        
    - Server Error
        
        ```yaml
        HTTP/1.1 500 Internal Server Error
        Content-Type: application/problem+json
        Content-Language: en
        
        {
          "type": "/problems/invalid-permission",// replace specific code
          "title": "Invalid permission",
          "status": 500,
          "detail": ".....",
          "errors": []
        }
        ```
        

## Example

Let’s take window’s error code as the example. See [https://learn.microsoft.com/zh-tw/windows-hardware/drivers/debugger/bug-check-code-reference2](https://learn.microsoft.com/zh-tw/windows-hardware/drivers/debugger/bug-check-code-reference2).

DO

```jsx
{
  "type": "/problems/ipt-watchdog-timeout",
  "status": 500,
  "detail": "...."
}
```

DON’T

```jsx
{
  "status": "0x0001db"
}
```
