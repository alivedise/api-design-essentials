# [ADP-115] PATCH

## Overview

The HTTP PATCH method is used to apply partial modifications to a resource. Unlike the PUT method, which replaces the entire resource, PATCH only updates specific fields or attributes.

## Use Cases

- **Partial Updates:** When only a portion of the resource needs to be updated, PATCH can be more efficient than PUT.
- **Large Resources:** For resources with a large amount of data, PATCH reduces the amount of data transferred.
- **Concurrent Updates:** Helps in scenarios where multiple clients might update different parts of a resource concurrently.

## Request Format

- **Endpoint:** The URL of the resource to be updated.
- **HTTP Method:** PATCH
- **Headers:**
    - `Content-Type`: Specifies the media type of the request body (e.g., `application/json-patch+json`, `application/merge-patch+json`).
    - `Authorization`: Authentication token if required.
- **Body:** Contains the instructions for the modifications, structured according to the specified media type.

## Response Format

- **Status Codes:**
    - `200 OK`: Successful modification, and the response contains the updated resource.
    - `204 No Content`: Successful modification with no response body.
    - `400 Bad Request`: The request is malformed or contains invalid data.
    - `401 Unauthorized`: Authentication is required and has failed or has not been provided.
    - `403 Forbidden`: The client does not have permission to modify the resource.
    - `404 Not Found`: The resource to be modified does not exist.
    - `409 Conflict`: The request could not be completed due to a conflict with the current state of the resource.
- **Headers:**
    - `Content-Type`: Specifies the media type of the response body.
- **Body:** Contains the updated resource if the status code is `200 OK`.

## Guidance

- MUST validate the patch document before applying it to the resource to ensure it is well-formed and contains no errors.
- SHOULD use a standardized format for the patch document, such as JSON Patch (`application/json-patch+json`) or Merge Patch (`application/merge-patch+json`).
- MUST ensure that only authorized users can perform PATCH operations to maintain security and data integrity.
- SHOULD handle concurrent modifications gracefully, potentially using mechanisms such as optimistic locking.
- MUST return appropriate HTTP status codes to indicate the outcome of the PATCH operation.
- SHOULD design PATCH operations to be idempotent, meaning that applying the same patch multiple times does not result in different outcomes.
- MAY include the updated resource in the response body if the status code is `200 OK`.

### Example

**Example 1: JSON Patch**

```jsx
PATCH /users/123 HTTP/1.1
Host: example.com
Content-Type: application/json-patch+json
Authorization: Bearer token

[
  { "op": "replace", "path": "/email", "value": "newemail@example.com" },
  { "op": "add", "path": "/phone", "value": "+1234567890" }
]
```

**Example 2: Merge Patch**

```jsx
PATCH /users/123 HTTP/1.1
Host: example.com
Content-Type: application/merge-patch+json
Authorization: Bearer token

{
  "email": "newemail@example.com",
  "phone": "+1234567890"
}
```

### Best Practices

1. **Validation:** Ensure the request data is validated before applying the patch to avoid corrupting the resource.
2. **Atomicity:** Apply all changes atomically to maintain the resource's integrity.
3. **Idempotence:** While PATCH is not inherently idempotent, strive to design your patch operations in a way that reapplying the same patch does not lead to inconsistent states.
4. **Documentation:** Clearly document the supported PATCH operations and the structure of the request body for clients.

### Security Considerations

- **Authentication:** Ensure that only authorized users can perform PATCH operations.
- **Validation:** Sanitize and validate input to prevent security vulnerabilities such as injection attacks.
- **Concurrency:** Implement mechanisms to handle concurrent updates to the same resource gracefully, such as optimistic locking.

# Accept-Patch

## Reference

- [https://www.rfc-editor.org/rfc/rfc5789#section-3.1](https://www.rfc-editor.org/rfc/rfc5789#section-3.1)
- https://developer.mozilla.org/zh-TW/docs/Web/HTTP/Headers/Accept-Patch
- [https://fullstack.wiki/http/headers/Accept-Patch](https://fullstack.wiki/http/headers/Accept-Patch)

## General

- This is a response only header.
- MAY use Accept-Patch to tell the client the requesting resource with PATCH method and `Content-Type` is not supported with 415 status. In the other hand, if the Content-Type is not specified in the request header, SHOULD NOT response with Accept-Patch nor 415.
    
    ```jsx
    PATCH /planet
    Content-Type: text/example
    
    415
    Accept-Patch: application/vnd.planet+json
    ```
    
- There’s another use case for `OPTIONS` request; to tell the client the requesting OPTION method on the resource is PATCH-able.
