# [ADP-130] Idempotent-Key

## Overview

Idempotency is a key concept in API design, ensuring that multiple identical requests have the same effect as a single request. The use of an Idempotency-Key allows clients to safely retry requests without unintended side effects. This is particularly useful for operations like POST and PATCH, which are not inherently idempotent. The following rules and examples illustrate how to implement and handle Idempotency-Key in your APIs.

## Guidance

- SHOULD support Idempotency-Key on demand for POST/PATCH.
    - APIs should allow clients to use Idempotency-Keys to ensure that multiple identical requests are processed only once.
- SHOULD make sure the Idempotency-Key is unique.
    - The server must ensure that the Idempotency-Key provided by the client is unique for each request to prevent duplicate processing.
- SHOULD use uuid for Idempotency-Key. (As RFC draft suggests).
    - Using UUIDs (as suggested by the RFC draft) ensures a high level of uniqueness and randomness for Idempotency-Keys.
- **The client has its choice to decide the request is with Idempotency-Key or not.**
    - The client can choose to include an Idempotency-Key in the request header. If the request fails due to network issues, the client can retry with the same Idempotency-Key. The server will check if the request has already been processed and return the previous result if it has.
- SHOULD response 400 Bad Request if an API is designed with Idempotency-Key as required field. (As RFC draft suggests)
    
    ```jsx
    HTTP/1.1 400 Bad Request
    Content-Type: application/problem+json
    Content-Language: en
    {
     "type": "https://developer.example.com/problems/missing-idempotency",
     "title": "Idempotency-Key is missing",
     "detail": "This operation is idempotent and it requires correct usage of Idempotency Key.",
    }
    ```
    

## Example

### Scenario

A client wants to create a new resource using a POST request. The client includes an Idempotency-Key in the request header to ensure that if the request is retried, the resource is not created multiple times.

### Request

```jsx
POST /resources HTTP/1.1
Host: api.example.com
Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000
Content-Type: application/json

{
  "name": "Example Resource",
  "type": "example"
}
```

### Server Processing

1. The server receives the request and checks the Idempotency-Key.
2. If the Idempotency-Key has not been used before, the server processes the request and creates the resource.
3. If the Idempotency-Key has been used, the server returns the result of the previous request.

### Response

```jsx
HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": "abc123",
  "name": "Example Resource",
  "type": "example"
}
```

### Example with OpenAPI

```yaml
openapi: 3.1.0
info:
  title: Example API
  version: 1.0.0
paths:
  /resources:
    post:
      summary: Create a new resource
      operationId: createResource
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              properties:
                name:
                  type: string
                type:
                  type: string
              required:
                - name
                - type
      responses:
        '201':
          description: Resource created successfully
          content:
            application/json:
              schema:
                type: object
                properties:
                  id:
                    type: string
                  name:
                    type: string
                  type:
                    type: string
        '400':
          description: Idempotency-Key is missing
          content:
            application/problem+json:
              schema:
                $ref: '#/components/schemas/Error'
      parameters:
        - name: Idempotency-Key
          in: header
          required: false
          schema:
            type: string
          description: A unique identifier to ensure idempotency

components:
  schemas:
    Error:
      type: object
      properties:
        type:
          type: string
        title:
          type: string
        detail:
          type: string
```
