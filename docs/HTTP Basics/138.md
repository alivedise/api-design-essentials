# [ADP-138] Rate Limit Headers

In the development and maintenance of APIs, it is critical to manage and communicate usage limits effectively to ensure fair and stable operation. One effective method is through the use of rate limited headers. This guideline outlines why and how rate limited headers SHOULD be utilized in API design, in compliance with RFC 9457.

## Why Use Rate Limited Headers?

1. **Fair Usage Enforcement**: Ensures that no single user or application can monopolize the API resources, providing a fair access opportunity for all users.
2. **Prevent Overloading**: Helps in protecting the backend infrastructure from being overwhelmed by too many requests in a short period, which can lead to performance degradation or downtime.
3. **User Experience**: Communicates to API consumers their current usage status and limits, helping them to manage their request rates and avoid unexpected service disruptions.
4. **Scalability**: Facilitates the scalability of the API by controlling the request rate, ensuring that the service can handle increasing numbers of users without compromising performance.

## Recommended Rate Limit Headers

The following headers SHOULD be implemented to communicate rate limits effectively:

1. **X-RateLimit-Limit**: Indicates the maximum number of requests allowed in a given time period.
2. **X-RateLimit-Remaining**: Shows the number of requests remaining in the current time period.
3. **X-RateLimit-Reset**: Specifies the time at which the rate limit counters reset, typically provided as a Unix timestamp.
4. **Retry-After**: Indicates how long the client SHOULD wait before making another request after exceeding the rate limit.

## Implementation Guidelines

1. **Consistent Application**: Rate limiting headers SHOULD be applied consistently across all API endpoints that are subject to rate limits.
2. **Documentation**: Clearly document the rate limiting policy, including the limits and the meaning of each header, in the API documentation.
3. **Error Handling**: When a rate limit is exceeded, the server SHOULD respond with a 429 (Too Many Requests) status code along with the Retry-After header to indicate when the client can retry the request.
4. **Granularity**: Define rate limits based on the specific needs and usage patterns of different endpoints or user tiers. This granularity can help optimize resource allocation and user satisfaction.
5. **Monitoring and Adjustment**: Regularly monitor the usage patterns and adjust the rate limits as necessary to ensure optimal performance and user experience.

## Error Response Format (RFC 9457)

When a rate limit is exceeded, the server SHOULD respond with a standardized error format as defined in RFC 9457. Below is an example of how to format such a response:

### Example Error Response

```jsx
HTTP/1.1 429 Too Many Requests
Content-Type: application/problem+json
Retry-After: 3600
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1625247600

{
  "type": "https://example.com/problems/rate-limit-exceeded",
  "title": "Rate Limit Exceeded",
  "status": 429,
  "detail": "You have exceeded your rate limit. Please wait until the specified time before making further requests.",
  "instance": "/users/12345",
  "rateLimit": {
    "limit": 1000,
    "remaining": 0,
    "reset": 1625247600,
    "retryAfter": 3600
  }
}
```

### Response Field Descriptions

- **type**: A URI reference that identifies the problem type.
- **title**: A short, human-readable summary of the problem type.
- **status**: The HTTP status code generated by the origin server for this occurrence of the problem.
- **detail**: A human-readable explanation specific to this occurrence of the problem.
- **instance**: A URI reference that identifies the specific occurrence of the problem.
- **rateLimit**: An object containing details about the rate limit, including:
    - **limit**: The maximum number of requests allowed.
    - **remaining**: The number of requests remaining in the current time period.
    - **reset**: The time at which the rate limit counters reset, typically provided as a Unix timestamp.
    - **retryAfter**: The time (in seconds) the client should wait before making another request after exceeding the rate limit.
