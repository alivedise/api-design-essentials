# [ADP-757] OpenAPI: x-violated-adp

If the implementation somewhat violates a specific rule, the developer MAY be expected to feedback the reason on their own document. For example, in OpenAPI 3.1, use extension in each operation to specify what and why certain api-principle-rule is violated.

The description inside extension would not be exposed to public after document generation.

```yaml
openapi: 3.1.0
info:
  title: Sample API
  version: 1.0.0
paths:
  /example:
    get:
      summary: Sample GET operation
      description: This operation may violate some adp.
      x-violated-rules:
        - ruleId: "api-principle-rule-00001"
          description: "The design violates rule 00001 because..."
        - ruleId: "api-principle-rule-00002"
          description: "Thie design violates rule 00002 because..."
      responses:
        '200':
          description: Successful response
```

<aside>
💡 If there’s a good reason, you are expected to feedback it to this principle to change it for the whole company.

</aside>
