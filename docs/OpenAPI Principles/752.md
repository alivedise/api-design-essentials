# [ADP-752] OpenAPI: API audience

## Overview

- API audience is an important field to identify the usage of the API and would impact the management of the APIs. There’re many principles are designed based on the audience of the API. In OpenAPI, we utilize the extension field for API audience as`x-audience`.

<aside>
💡 **Some api principle rules are based on the audience.**

</aside>

<aside>
💡 Public documentation site are generated based on the audience.

</aside>

### OpenAPI definition for x-audience

```bnf
#/info/x-audience:
  type: string
  x-extensible-enum:
    - COMPONENT_INTERNAL
    - BUSINESS_UNIT_INTERNAL
    - COMPANY_INTENRAL
    - PARTNER_EXTERNAL
    - PUBLIC_EXTERNAL
  description: |
    Intended target audience of the API. Relevant for standards around
    quality of design and documentation, reviews, discoverability,
    changeability, and permission granting.
```

## Guidance

- MUST provide x-audience, which are one of the following order by audience size
    
    
    | Value | Description |
    | --- | --- |
    | COMPONENT_INTERNAL | This means the API usage is team specific. |
    | BUSINESS_UNIT_INTERNAL | This means the API usage is under the same business unit. |
    | COMPANY_INTERNAL | This means the API audience is targeting different business units within the company. |
    | PARTNER_EXTERNAL | This means the API audience is specific partners. |
    | PUBLIC_EXTERNAL | This means the API audience is everyone who has the right authentication. |

## Example

```markdown
openapi: 3.0.1
info:
  title: Sample API
  description: This is a sample API to demonstrate the usage of the x-audience field.
  version: 1.0.0
  contact:
    name: API Support
    url: http://www.example.com/support
    email: support@example.com

paths:
  /items:
    get:
      summary: Retrieve a list of items
      operationId: getItems
      x-audience: PUBLIC_EXTERNAL// [!code highlight]
      responses:
        '200':
          description: A list of items
          content:
            application/json:
              schema:
                type: array
                items:
                  type: object
                  properties:
                    id:
                      type: integer
                      example: 1
                    name:
                      type: string
                      example: Item 1

  /internal/items:
    get:
      summary: Retrieve a list of internal items
      operationId: getInternalItems
      x-audience: COMPONENT_INTERNAL// [!code highlight]
      responses:
        '200':
          description: A list of internal items
          content:
            application/json:
              schema:
                type: array
                items:
                  type: object
                  properties:
                    id:
                      type: integer
                      example: 1
                    name:
                      type: string
                      example: Internal Item 1

components:
  schemas:
    Item:
      type: object
      properties:
        id:
          type: integer
        name:
          type: string

  responses:
    NotFound:
      description: Resource not found
    Unauthorized:
      description: Unauthorized access

  securitySchemes:
    ApiKeyAuth:
      type: apiKey
      in: header
      name: X-API-Key

security:
  - ApiKeyAuth: []
```

## Design Thinking

- When designing APIs, it is important to consider the audience priority. For instance, an API designed for `PUBLIC_EXTERNAL` usage can be utilized by an internal team. However, there are scenarios where it is necessary to design APIs for different audiences. In such cases, it is crucial to design them carefully and assign unique `operationId` values for each API to distinguish their purposes and ensure proper usage. See [SHOULD provide operationId](755.md).
    
    > ***Note:** Exactly one audience per API specification is allowed. For this reason a smaller audience group is intentionally included in the wider group and thus does not need to be declared additionally. If parts of your API have a different target audience, we recommend to split API specifications along the target audience — even if this creates redundancies (rationale).*
