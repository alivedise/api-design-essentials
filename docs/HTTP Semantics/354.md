# [ADP-354] Client Hints

## Overview

Client Hints is designed to provide the device information and let the server to decide the best choice for the requested client. It’s for proactive(client-driven) content negotiati

Client Hint would be a replacement for the original User-Agent in a more elegant way. By using Client Hints, servers can optimize the content delivery based on the client's actual hardware and network conditions, leading to a more efficient and tailored user experience.

## Client Hints Infrastructures

### Low entropy hints

[`Save-Data`](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/Save-Data)、[`Sec-CH-UA`](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/Sec-CH-UA)、[`Sec-CH-UA-Mobile`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Sec-CH-UA-Mobile)、[`Sec-CH-UA-Platform`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Sec-CH-UA-Platform)

### High entropy hints

All CH which is not low entropy client hints.

If the server is designed to interact with the client hint, it should return `Accept-CH` in the response header. However, this requires an extra request. Alternatively, in a modern browser that supports client hints, we can include a meta tag in the document header to skip the first request for the server response with `Accept-CH`.

If the server's `Accept-CH` has different content from the meta tag, the browser combines the lists into one.

```jsx
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <meta http-equiv="Accept-CH" content="DPR, Viewport-Width, Width, Save-Data, Downlink, RTT, Sec-CH-UA-Full-Version, Sec-CH-UA-Platform-Version, Sec-CH-UA-Model">
    <title>Client Hints Test</title>
</head>
```

Note: the browser might limit the information about high entropy client hints. Make your API design adopt this.

Note: the browser is expected to send HTTP request with low entropy client hints withouting specifying in `Accept-CH` .

Note: `Save-Data` is defined by the user preference. See https://web.dev/articles/optimizing-content-efficiency-save-data

## Guidance

- SHOULD Improve user experience by adapting content based on device capabilities
- SHOULD enhance performance by reducing unnecessary data transfer
- SHOULD design for incremental client hints
    - The client might limit the information sent by client hints; the API shall react to the empty value of client hints.
- SHOULD know the privacy concern of high entropy client hints
    - Use low entropy client hints at first.
    - If the API is built with web page, MAY implement a client hint preference UI to let the user choose what information to send. The server should respect the preference and limit the usage of `Accept-CH` .

## Example

### Design an API which response would vary per client hints

```jsx
GET /api/chart-data HTTP/1.1
Host: api.example.com
DPR: 2.0
```

```jsx
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 2048

{
  "chart": "data with high resolution"
}
```

---

```jsx
GET /data HTTP/1.1
Host: api.example.com
Downlink: 0.5
RTT: 100
```

```jsx
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 1024

{
  "data": "subset of full data due to slow network"
}
```

---

```jsx
GET /articles/12345 HTTP/1.1
Host: api.example.com
Save-Data: on
```

```jsx
{
  "id": "12345",
  "title": "Breaking News: Example Event",
  "summary": "A brief summary of the news event.",
  "content": "This is the full version of the article content, with detailed descriptions and multimedia...",
  "images": [
    "https://example.com/images/1.jpg",
    "https://example.com/images/2.jpg"
  ],
  "videos": [
    "https://example.com/videos/1.mp4"
  ]
}
```
