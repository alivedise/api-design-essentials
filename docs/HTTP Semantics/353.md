# [ADP-353] Content Negotiation

## Overview

![Untitled](%5BADP-353%5D%20Content%20Negotiation%2072a4a54c3b5e4e79bc15bbd94e967917/Untitled.png)

Content negotiation is a mechanism that allows a server to select the best representation of a resource based on the requested format by a client. It enables different versions of a response (like HTML, XML, JSON, etc.) for the same resource. It elevates the API with content accessibility.

For instance, when a client sends a request to a server, it includes headers to tell the server what formats it accepts. These may include text/html, application/json, application/xml, etc. The server then responds with a representation of the resource that best matches the client's preferences, if available.

![Untitled](%5BADP-353%5D%20Content%20Negotiation%2072a4a54c3b5e4e79bc15bbd94e967917/Untitled%201.png)

The main types of content negotiation are:

1. Proactive(Server-driven): The server selects the representation based upon the client's stated preferences.
2. Reactive(Client-driven, or Agent-driven): The client specifies the representation in the request.

The most commonly used method is server-driven negotiation, where the Accept header is used by HTTP clients to tell the server what content types they'll accept.

The benefits of supporting content negotiation include flexibility and interoperability. It allows clients and servers to evolve independently, as new formats can be added without breaking existing clients.

<aside>
💡 SHOULD NOT tackle client-driven negotiation unless you document the progress of negotiation well.

</aside>

The reason is specified in MDN; briefly, it’s due to there’s no standard about how to negotiate after the first trail of client → server.

## Guidance

- SHOULD return `application/json` compatible MIME type according to RFC

## Relative HTTP headers

- Request
    - Accept: MIME Type
        - Possible values
            - images/jpeg
            - application/json
            - application/vnd.vvtk.alive-app-1.user.v1+json
        - While accepting json compatible MIME types, the client is expected to specify the resource type by extending as above. See MUST follow MIME type naming convention.
        - The client is expected to request a list of accepting MIME types separated by comma or a q-factor list
            
            ```jsx
            Accept: application/json, images/jpeg
            ```
            
    - Accept-Encoding
        - Possible values
            - gzip
            - compress
            - deflate
            - br
            - identity
        - The client is able to send a wish list by q-factor list such as
            
            ```jsx
            Accept-Encoding: gzip;q=1.0, identity;q=0.5, *;q=0
            ```
            
            The request header specifies the client wants gzip more; the second priority is identity, and doesn’t expect any other encodings.
            
    - Accept-Charset
        - MUST NOT use this header. See https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Headers/Accept-Charset
            - The client SHOULD NOT specify this header
            - The server MUST ignore this header
    - Accept-Language
        - According to [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Content_negotiation#the_accept-language_header), we SHOULD NOT rely on this header.
        - SHOULD design another specific API for i18n preference.
    - Accept-CH
        - Used for [client hints](https://developer.mozilla.org/en-US/docs/Web/HTTP/Client_hints); to tell the server to cache certain client hint in a specific life time for requests afterwards
        
        ```jsx
        Accept-CH: Width
        ```
        
    - User-Agent
        - MUST NOT use user-agent info for content negotiation. It’s a bad practice to guess the user environment from the user agent value.
- Response
    - Content-Type: MIME types
        - For common MIME types, SHOULD return the specified if supported
        - For custom MIME types, check MUST follow custom MIME type naming convention.
        - If the specified common MIME type is not supported, SHOULD choose the appropriate return type for the client.
        - If the specified custom MIME type is not supported, SHOULD return 406 Not acceptable.
        - If the specified custom MIME type is deprecated, SHOULD set `DEPRECATE` in response header. See SHOULD consider API life cycle management.
    - Content-Encoding
        - Possible values: same as request
        - SHOULD return `identity` if the server cannot find a suitable encoding per the client request.
    - Language
        - Since we discourage to use `Accept-Language` for negotiating the user language, we SHOULD always return `Language: en` if the response involve human readable context such as error message.
    - Vary
        - The value would be respected by cache proxy. SHOULD specify Vary for cache control while the server is accepting the client’s accept-* request.
        
        <aside>
        💡 Use `Vary: *` to specify the response is uncacheable.
        
        </aside>
        
        - In HTTP caching, it’s basically based on URL. However, we wouldn’t expect to cache content-negotiated result because it may give us unexpected response. The `Vary` header acts just like its name, to tell the cache proxy to cache the response based on the “Varying” negotiation. See [HTTP Caching - Vary in MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching#vary).

<aside>
💡 SHOULD NOT use User-Agent to do content negotiation

</aside>

This discussion comes from [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Browser_detection_using_the_user_agent)

## Example

The following examples exhibit the possible content negotiation for API design.

### Design an API to fetch assets by url or image type on demand.

```jsx
GET /assets/asset-id HTTP/1.1
Accept: image/jpeg

HTTP/1.1 200 OK
Content-Type: image/jpeg
(binary)
```

```jsx
GET /assets/asset-id HTTP/1.1
Accept: application/json

HTTP/1.1 200 OK
Content-Type: application/json

{
  url: '....',
}
```

```jsx
GET /assets/asset-id HTTP/1.1
Accept: image/webp

HTTP/1.1 200 OK
Content-Type: image/webp
(binary)
```

### Design an API to adopt different modifier to the same resource

```jsx
GET /api-principles/1
Accept: application/vnd.my-corp.alive-app.api-principle.simplified+json

200
Content-Type: application/vnd.my-corp.alive-app.api-principle.simplified+json

{
  title: 'SHOULD adopt best practices',
  principleId: 1
}
```

```jsx
GET /api-principles/1
Accept: application/vnd.my-corp.alive-app.api-principle+json

200
Content-Type: application/vnd.my-corp.alive-app.api-principle+json

{
  title: 'SHOULD adopt best practices',
  principleId: 1,
  examples: [],
  references: [],
  designThing: '....'
}
```
