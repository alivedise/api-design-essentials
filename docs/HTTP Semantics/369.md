# [ADP-369] HTTP Caching: ETag

## Overview

### ETag

The `ETag` header is used for resource versioning. It provides a way to validate the cache.

### Usage

- The server generates a unique ETag for each version of the resource.
- Clients include the ETag in the `If-None-Match` header to validate the cached resource.
    - Notice: the browser would attach `If-None-Match` automatically. If the API caller is not inside the browser, it should attach `If-None-Match` when it sees the `ETag` in the API response.

### Example

```json
ETag: "abc123"
If-None-Match: "abc123"
```

## Procedures

1. The client initialize the first GET request without any cache related header.
2. The server returns the data with calculated `Etag`.
3. The client requests the same GET again with `If-None-Match`.
4. The server checks the `If-None-Match` request with latest data’s `Etag`.
    1. return 200 if mismatch
    2. return 304 if matched

## Guidance

- MAY supply ETag to cache the response

## Implementation Detail

- There’re two kind of ETag: `Weak ETag` and `Strong ETag` .
- In `express` (nodejs web server framework), it would generate weak Etag by default; could generate strong ETag by changing the configuration
    - weak ETags are generated using [CRC32](http://en.wikipedia.org/wiki/Cyclic_redundancy_check) ([source](https://github.com/visionmedia/express/blob/3ffceff3ed356a711fce5fee6ed00a03f43affd8/lib/utils.js#L53)) and strong ETags are generated using MD5 ([source](https://github.com/visionmedia/express/blob/3ffceff3ed356a711fce5fee6ed00a03f43affd8/lib/utils.js#L28)).
- In `gin`(golang web server framework), ETag could be generated via a middleware.
