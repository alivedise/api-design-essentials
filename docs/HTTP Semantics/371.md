# [ADP-371] HTTP Caching: Cache-Control

## Overview

### Cache-Control

The `Cache-Control` header provides directives for caching mechanisms in both requests and responses.

### Common Directives

- **public**: Indicates that the response may be cached by any cache.
- **private**: Indicates that the response is intended for a single user(the brower client) and should not be cached by shared caches(Proxies, CDNs)
- **no-cache**: Forces caches to submit the request to the origin server for validation before releasing a cached copy.
- **no-store**: The response should not be stored in any cache.
- **max-age=`<seconds>`**: Specifies the maximum amount of time a resource will be considered fresh.
- **must-revalidate**: Indicates that once a resource becomes stale, it must be validated with the origin server before use.

### Example

```json
Cache-Control: public, max-age=3600, must-revalidate
```

## Guidance

- SHOULD use Cache-Control for HTTP Caching
- SHOULD use `private` directive to avoid being cached by shared cachies.

## Design Thinking

- The mose use case for API is to cache. the binary response such as `image/jpeg`
