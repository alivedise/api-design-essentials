# [ADP-1] ADP Purpose

## Goals

- One of the goals of this document is to establish a standard that enables every API designer in the company to create APIs that appear as if they were designed by the same individual.
- Ensure that all API designs adhere to a unified set of best practices and guidelines, promoting consistency and quality across all projects.
- Facilitate better collaboration among teams by providing a common language and framework for API design, making it easier to understand and integrate APIs across different business units.
- Ensure that the API designs prioritize both the developer experience and the end-user experience, leading to more robust and user-friendly applications.

## Principle of Principle

- **Product Agnostic**: The principles are designed to be product-agnostic and can be applied to every product, project, or application across business units within the same company.
- **Rule and Suggestion Structure**: Each principle may include one or more rules or design suggestions.
- **RFC Requirement Keywords**: Each rule must use `MUST`, `MUST NOT`, `SHOULD`, `SHOULD NOT`, `MAY`, or `MAY NOT` in uppercase, following [RFC 2119 requirement keywords](https://datatracker.ietf.org/doc/html/rfc2119).

## Descriptions

- **Inclusion of Keywords**: Descriptions containing these uppercase keywords are considered part of the rule.
- **Complex Rules**: If a rule requires more than one sentence for description, it should become a new ADP page with a unique ID recorded in the markdown’s front matter area. This ID is used for web page routing and communication between developers. <aside> 💡 The naming convention `incrementing-number-based-id` is preferred over UUIDs due to maintenance considerations. This approach simplifies tracking and managing the identifiers, making it easier to understand and maintain over time. </aside>
    
    ```markdown
    ---
    id: 1
    ---
    
    # MUST treat the developer experience as important as the user experience
    ## Overview
    ...
    ```
    

## Violating Rules

- **Specifying Violations**: If you need to violate certain rules, you [MAY specify violated rules](757.md).
    
    <aside>
    💡 If there’s a good reason for the violation, you are expected to provide feedback to this principle to consider changes for the whole company.
    
    </aside>
    

## Draft Rules

- **Internal API Portal**: There is an internal API portal used for application, problem, and event registration, inspired by IANA registration.
- **Draft Status**: Use `[DRAFT]` before a rule to indicate that it is not yet stable.

## Proposing a New ADP

- **Format**: Use markdown format.
- **Submission**: File a pull request in the ADP site’s repository. (Note: it might be a mirrored repository. Find out the upstream repository).
