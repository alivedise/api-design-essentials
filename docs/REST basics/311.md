# [ADP-311] Filtering

## Overview

In RESTful APIs, filtering is a common requirement to allow clients to retrieve only the data that meets specific criteria. This guideline provides a standard approach for implementing filtering in RESTful APIs, focusing on the use of nested attributes and other common methods.

## Guidance

- [DRAFT] SHOULD support filtering by common `filter` query parameter
    
    ```jsx
    GET /users?filter.name=John HTTP/1.1
    ```
    
- [DRAFT] SHOULD support value comparison by common keywords
    
    Comparison operators can be used to specify ranges or conditions. Common operators include:
    
    - `eq` (equal)
    - `ne` (not equal)
    - `gt` (greater than)
    - `gte` (greater than or equal)
    - `lt` (less than)
    - `lte` (less than or equal)
    
    ```jsx
    GET /users?filter.releasedAt:gt=2024-07-13T00:00:00 HTTP/1.1
    ```
    
- [DRAFT] SHOULD support nested filter
    
    ```jsx
    GET /users?filter.address.region=eu HTTP/1.1
    ```
    
- SHOULD support value OR by `,`
    
    ```jsx
    GET /users?filter.role=MANAGER,SUPERVISOR
    ```
    

## Design Thinking

### Why not property only design?

For result filtering, a possible design is using the field name only such as

```jsx
GET /emaps?region=us&category=self-defined
```

The key issue of this way is it might conflict with common query parameters. If you’d like to implement more custom query parameters, it’s more often to collide.

### Why not brackets?

One possible notation with `filter` from JSONAPI is to use `[]` to represent the JSON field. However `[]` might be encoded in the URL and looks non-intuitive.

```jsx
const baseUrl = 'https://api.example.com/articles';

const params = {
  'author[eq]': 'chris',
  'status[ne]': 'past',
  'count[gte]': '200'
};

const queryString = new URLSearchParams(params).toString();

const url = `${baseUrl}?${queryString}`;

fetch(url)
  .then(response => {
    if (!response.ok) {
      throw new Error('Network response was not ok ' + response.statusText);
    }
    return response.json();
  })
  .then(data => {
    console.log('Fetched data:', data);
  })
  .catch(error => {
    console.error('There was a problem with the fetch operation:', error);
  });
```

The request above results in this in Network:

```jsx
'https://api.example.com/articles?author%5Beq%5D=chris&status%5Bne%5D=past&count%5Bgte%5D=200'
```

### LHS/RHS

LHS (left-hand side) means that we put the logic on the left-hand side of the `=` sign. RHS is for the right-hand side.

- Left Hand Side brackets
    
    ```jsx
    filter[key1][gt]=x
    ```
    
- Right Hand Side brackets
    
    ```jsx
    filter[key1]=[gt]x
    ```
    
- Left Hand Side colon
    
    ```jsx
    filter.key:gte=y
    ```
    
- Right Hand Side colon
    
    ```jsx
    filter[key]=gte:x
    ```
    

Using RHS would be confusing since it affects the analytics of the value, which should not be restricted.

Using LHS would be preferable since we already limit the property of the JSON to be a camelCase string without `:`, `[`, `]`, or `.`.
