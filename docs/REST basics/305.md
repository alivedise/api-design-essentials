# [ADP-305] /api/ in relative uri path

## Guidance

- SHOULD NOT have `/api/` in the relative URL Path (excluding the domain).

To maintain a clean and intuitive URL structure, it is recommended that the URL paths for APIs SHOULD NOT include the segment `/api/`. This approach adheres to the principle of simplicity and ensures that the URLs are user-friendly and easy to understand.

It’s acceptable to use an `api` subdomain.

**Exception**: If your web server cannot provide web pages and APIs on different domains, you may include `/api/` in the path to differentiate.

## Rationale

1. **Readability and Simplicity**:
    - Avoiding `/api/` in the URL path keeps the endpoint concise and straightforward. For instance, `https://example.com/users` is clearer and more direct than `https://example.com/api/users`.
2. **Versioning and Flexibility**:
    - Excluding `/api/` allows for more flexibility in managing versions and other aspects of the API without being tied to a specific segment. Versioning can be handled through other means, such as headers or query parameters.
3. **Consistency**:
    - Consistent URL structures are easier for developers to remember and use. By not including `/api/`, all resources can follow a uniform pattern, enhancing the developer experience.
4. **Separation of Concerns**:
    - The URL path should reflect the resource hierarchy and relationships rather than indicating the technical implementation (such as `/api/`). This helps in maintaining a clear separation between the API design and its underlying technology.

### Example

**Recommended:**

- `https://api.example.com/users`
- `https://api.example.com/orders`
- `https://api.example.com/products`

**Not Recommended:**

- `https://example.com/api/users`
- `https://example.com/api/orders`
- `https://example.com/api/products`
