# [ADP-315] REST maturity

### Overview

The RESTful API Maturity Model, proposed by Roy Fielding, measures the extent to which an API adheres to REST principles. The model is divided into four levels: Level 0 to Level 3. This document outlines the requirements for APIs to achieve at least Level 2 maturity and explains their significance.

### Maturity Level Overview with Examples

1. **Level 0: The Swamp of POX (Plain Old XML)**
    - Single endpoint. Uses HTTP as a transport protocol. Typically, RPC-style endpoints.
    - **Example**:
        
        ```jsx
        POST /api HTTP/1.1
        Host: example.com
        Content-Type: application/xml
        
        <request>
          <action>getUser</action>
          <userId>123</userId>
        </request>
        ```
        
        ```jsx
        HTTP/1.1 200 OK
        Content-Type: application/xml
        
        <response>
          <user>
            <id>123</id>
            <name>John Doe</name>
          </user>
        </response>
        ```
        
2. **Level 1: Resources**
    - Multiple endpoints representing different resources. Each resource has its own URI. Still uses HTTP as a transport protocol, often without full use of HTTP methods.
    - **Example**:
        
        ```jsx
        GET /users/123 HTTP/1.1
        Host: example.com
        ```
        
        ```jsx
        HTTP/1.1 200 OK
        Content-Type: application/json
        
        {
          "id": 123,
          "name": "John Doe"
        }
        ```
        
3. **Level 2: HTTP Verbs**
    - Full use of HTTP methods (GET, POST, PUT, DELETE). Utilizes HTTP status codes for responses. Supports content negotiation (e.g., JSON, XML).
    - **Example**:
        - **Create a User**:
            
            ```jsx
            POST /users HTTP/1.1
            Host: example.com
            Content-Type: application/json
            
            {
              "name": "John Doe"
            }
            
            ```
            
            ```jsx
            HTTP/1.1 201 Created
            Content-Type: application/json
            Location: /users/123
            
            {
              "id": 123,
              "name": "John Doe"
            }
            ```
            
        - **Update a User**:
            
            ```jsx
            PUT /users/123 HTTP/1.1
            Host: example.com
            Content-Type: application/json
            
            {
              "name": "John Smith"
            }
            ```
            
            ```jsx
            HTTP/1.1 200 OK
            Content-Type: application/json
            
            {
              "id": 123,
              "name": "John Smith"
            }
            ```
            
        - **Delete a User**:
            
            ```jsx
            DELETE /users/123 HTTP/1.1
            Host: example.com
            ```
            
            ```jsx
            HTTP/1.1 204 No Content
            ```
            
4. **Level 3: Hypermedia Controls**
    - Implements HATEOAS (Hypermedia As The Engine Of Application State). API responses include hyperlinks to related resources and actions. Provides discoverability through hypermedia.
    - **Example**:
        - **Get User with Links**:
            
            ```jsx
            GET /users/123 HTTP/1.1
            Host: example.com
            ```
            
            ```jsx
            HTTP/1.1 200 OK
            Content-Type: application/json
            
            {
              "id": 123,
              "name": "John Doe",
              "links": {
                "self": "/users/123",
                "friends": "/users/123/friends",
                "posts": "/users/123/posts"
              }
            }
            ```
            
        - **Get User's Friends**:
            
            ```jsx
            GET /users/123/friends HTTP/1.1
            Host: example.com
            ```
            
            ```jsx
            HTTP/1.1 200 OK
            Content-Type: application/json
            
            [
              {
                "id": 124,
                "name": "Jane Doe",
                "links": {
                  "self": "/users/124"
                }
              },
              {
                "id": 125,
                "name": "Jim Beam",
                "links": {
                  "self": "/users/125"
                }
              }
            ]
            ```
            

## Guidance

- MUST achieve Level 2 Maturity
- MAY provide hypermedia info in certain cases but not all, See [SHOULD support Resource Expansion](360.md).

## Design Thinking

To ensure that APIs achieve at least Level 2 maturity, the following guidelines MUST be followed:

1. **Resource Identification through URIs**
    - Each resource must have a unique URI.
    - URIs should be logical and consistent.
2. **Utilization of HTTP Methods**
    - See [HTTP Methods](110.md)

Additionally, the original definition does not cover but still importand: headers, status code and query parameters are also important when designing an API

- See [HTTP Headers](120.md), [HTTP Status](200.md), Query Parameters
