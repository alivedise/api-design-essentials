# [ADP-41] API Versioning

## Guidance

- MUST versioning your API endpoints which `audience≥PARTNET_EXTERNAL` .
- SHOULD choose from the common versioning strategies and document it well
- [MUST use semver](24.md)
- SHOULD avoid break change if possible
- SHOULD NOT remove field unless you are ready to increment major version number
- SHOULD consider to support new/old versions at the same time of the API which `audience>=PARTNET_EXTERNAL`.
- [DRAFT] We prefer uri versioning because it is clear in URL; but content negotiation via versioning might provide more feasibility.

## Design Thinking

There are several mechanisms for API versioning, each with its own use cases and benefits:

1. **URL Versioning**
    - **Domain Versioning**: This approach involves changing the domain to reflect the version. For example, `api.v1.example.com` and `api.v2.example.com`. This method can be useful for major version changes but might complicate DNS management and increase maintenance overhead.
    - **URI Versioning**: This is a common method where the version is included in the path. For example, `example.com/api/v1/resource`. It is straightforward and easy to implement and understand.
2. **Query Parameter Based Versioning**
    - This method involves adding a version parameter to the query string. For example, `example.com/api/resource?version=1`. This can be useful for optional versioning and testing new versions without changing the base URL.
3. **Header Based Versioning**
    - Version information is included in the HTTP headers. For example, by adding a custom header `API-Version: 1`. This method keeps the URL clean and is flexible but may not be as visible to users and developers.
4. **Resource based(Content-Type, MIME) Versioning**
    - The version is specified in the `Content-Type` header. For example, `Content-Type: application/vnd.example.v1+json`. This method is useful for APIs that support multiple content types and versions simultaneously.

### Common Values for Versioning

1. **Semver Based**: Using semantic versioning with major and minor version numbers, such as `v1` or `v1.1`.
2. **Date Based**: Using dates to indicate versioning, such as `2024-07-08`.
    
    <aside>
    💡 **Note**: Date-based versioning still follows semantic versioning principles. The date represents the release date of a major version, not the modification date of the API. If an API release does not include breaking changes to its schema, the date versioning value should remain unchanged.
    
    </aside>
    

Please note that versioning strategy is subject to change during maturity of API development.
