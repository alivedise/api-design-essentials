# [ADE-306] URI Design Patterns

## Overview

Designing paths for APIs is a task that does not have a completely correct answer and is prone to design disagreements. This specification attempts to provide some observed design patterns, which may improve over time. We encourage developers to share effective URI design examples they encounter (see [ADE-307](307)) to help refine these guidelines.

## Guidance

- SHOULD use lowercase letters for URI paths.
- SHOULD use hyphens (-) to separate words in URI paths (i.e., use kebab-case).
- MUST use forward slashes (/) to indicate hierarchy.
- SHOULD shorten URIs as much as possible while maintaining clarity.
- MUST NOT include file extensions (e.g., .json, .xml) in URIs.
- SHOULD use query parameters for filtering, sorting, and pagination.
  :::info
  This means that paths should not (SHOULD NOT) be used for these three tasks.
  For the three major collection operations, see [ADE-311](311), [ADE-312](312), [ADE-313](313).
  :::
- MUST encode special characters in URIs according to [RFC 3986](https://datatracker.ietf.org/doc/html/rfc3986).
- MUST NOT end URIs with a trailing slash (/).
  :::info
  Since the last character in a URI path does not add semantic value and may cause confusion, URIs must not end with a trailing slash (/).
  :::
  Correct practice

  ```raw
  /customers/{user-id}/addresses/{address-id}
  ```

  Incorrect practice

  ```raw
  /customers/{user-id}/addresses/{address-id}/
  ```

- SHOULD refer to the path patterns mentioned below for design.
- If needed, MAY include API version in the URI (e.g., `/v1/users`). See [ADE-41: API Versioning](41).

## Path Design Patterns

### Pattern 1: Based on Collections

**Basic URI for Collections**:

```http
/{resources}
```

Example:

```http
GET /users HTTP/1.1
```

### Pattern 1.1: Specific Resource in a Collection

**URI to access a specific resource in a collection**:

```http
/{resources}/{resource-id}
```

Example:

```http
GET /users/12345 HTTP/1.1
```

### Pattern 1.1B: Specific Resource in a Collection without ID Field

In some cases, it is necessary to access resources without a traditional `id` identifier. In this case, we must determine a unique field, such as `name` or `type`, to include in the URI as a reference.

**URI to access a resource without ID**:

```http
/{resources}/{resource-type|resource-name}
```

Example:

```http
GET /apis/{api-id}/stages/dev HTTP/1.1
```

This example demonstrates accessing a specific stage (`dev`) of the API identified by `{api-id}`, where the stage does not have a traditional `id` but can be uniquely identified by its `name`.

#### Guidelines for Resources without ID

1. **Unique Field Identification**: Ensure the selected field (e.g., `name` or `type`) is unique in the context used.
2. **Consistency**: Consistently use the same unique field across similar types of resources to avoid confusion.
3. **Clarity**: The selected field should clearly convey what resource it represents, making the URI easy to understand.

### Pattern 1.2: Nested Resources

**URI to access nested resources**:

```http
/{resources}/{resource-id}/{nested-resources}/{nested-resource-id}
```

Example:

```http
GET /users/12345/orders/67890 HTTP/1.1
```

### Pattern 1.3: Singleton Resources

Some resources naturally exist as single instances within a specific context, such as user profile settings or account configurations. These are called singleton resources.

**URI for singleton resources**:

```http
/{parent-resource}/{parent-id}/{singleton-resource}
```

Example:

```http
GET /users/12345/profile HTTP/1.1
GET /organization/67890/settings HTTP/1.1 # the settings here does not mean the plural form
```

#### Guidelines for Singleton Resources

1. **Use Singular Form**: Since the resource represents a single instance, use the singular form of the noun (e.g., `profile` not `profiles`).
2. **Context Clarity**: The singleton resource should make sense in its parent context.
3. **Consistency**: Maintain consistent naming across similar singleton resources.

### Pattern 2: Based on Actions

In RESTful API design, HTTP POST typically represents `CREATE`. However, many actions do not fully conform to standard CRUD operations. To address this, use an action verb to represent the operation. These URIs should always use `POST` as the HTTP method.

**Example Action URIs:**

```http
POST /search
POST /users/login
POST /documents/12345/publish
```

:::tip
HTTP methods SHOULD NOT be placed in the path, such as `POST /documents/delete`.
:::

#### Multi-word Action

For composite or action-based operations, use a subresource or action noun after the resource ID, following the pattern `/resources/${id}/${noun}-${verb}`. This is inspired by Google API's `resource:verb` convention, but uses `-` as the separator.

**Example Action URIs:**

```http
POST /users/123/password-reset
```

### Pattern 3: Resources with Modifiers

Modifiers can be used to filter or sort collections. This pattern should be intuitive and self-explanatory.

**Basic URI with Modifiers**:

```http
/{resources}/{modifier}
```

Example:

```http
GET /songs/recently-played HTTP/1.1
GET /articles/top-rated HTTP/1.1
```
