# [ADE-404] validation-error design

With Problem-Type `validation-error`, the server is expected to return multiple error instances regarding the failed part.

- All validation error rules and formats **MUST follow the OpenAPI specification's `format` field** (such as email, uuid, date-time, etc.) to ensure consistency and predictability.
- MUST provide error object info array for validation errors under `validationErrors` key

## Design Rationale

The `validationErrors` key is used instead of a generic `errors` key to clearly separate validation errors (input validation failures) from other types of errors. This explicit naming:

- Makes the response self-documenting
- Provides clear semantic meaning for API consumers

## Schema of a single validation error object

```json
[
  {
    "in": "[path|query|header|body]",
    "path": "#/json/pointer/to/failed/field",
    "invalidValue": "Optional invalid client input as a string.",
    "details": [
      {
        "key": "machine.readable.key1",
        "message": "First human-readable error message."
      },
      {
        "key": "machine.readable.key2",
        "message": "Another human-readable error message."
      }
    ]
  }
]
```

## OpenAPI 3.1 Example

Here's an example of how to define this error structure in OpenAPI 3.1:

```yaml
components:
  schemas:
    ValidationError:
      type: object
      required:
        - in
        - path
        - detail
      properties:
        in:
          type: string
          enum: [path, query, header, body]
        path:
          type: string
        detail:
          type: string
        invalidValue:
          type: string
        details:
          type: array
          items:
            type: object
            required:
              - key
              - message
            properties:
              key:
                type: string
              message:
                type: string
    
    ValidationErrorResponse:
      type: object
      required:
        - type
        - title
        - status
        - validationErrors
      properties:
        type:
          type: string
          const: 'validation-error'
        title:
          type: string
        status:
          type: integer
          enum: [400]
        validationErrors:
          type: array
          items:
            $ref: '#/components/schemas/ValidationError'

paths:
  /example:
    post:
      responses:
        '400':
          description: Bad Request
          content:
            application/problem+json:    
              schema:
                $ref: '#/components/schemas/ValidationErrorResponse'
```

This OpenAPI 3.1 example defines the validation error structure and shows how to use it in a response for a 400 Bad Request error.
